# Бизнес-процесс «Поиск и создание брони отеля»

## Этапы процесса

| Этап | Действие | Роль | Результат |
|:---|:---|:---|:---|
| 1. Инициация | Получение запроса от клиента: направление, даты заезда/выезда, количество гостей, пожелания по классу отеля и типу питания | Менеджер | Параметры поиска |
| 2. Поиск | Ввод параметров в систему. Система опрашивает пул подключённых Поставщиков по API или выгружает данные из квот и GDS/XML-фидов | Система | Список доступных вариантов |
| 3. Сравнение | Система выводит предложения с финальной ценой для туроператора (нетто + наценка + комиссия). Менеджер сравнивает варианты по цене, локации, условиям отмены | Менеджер | Выбранный вариант |
| 4. Предварительный расчёт | Система автоматически рассчитывает стоимость для клиента: базовая цена + наценка туроператора + дополнительные услуги (трансфер, страховка) | Система | Итоговая цена для клиента |
| 5. Холдирование (Pre-booking) | Менеджер нажимает «Забронировать». Система отправляет запрос провайдеру на временную блокировку номера | Система | Бронь в статусе «В обработке» |
| 6. Подтверждение или отказ | Ожидание ответа поставщика. Возможны три сценария (см. ниже) | Система / Менеджер | Изменение статуса брони |
| 7. Выписка документов | При статусе «Подтверждено» и внесении оплаты менеджер генерирует Ваучер (PDF) из карточки брони | Менеджер | Ваучер для клиента |

## Сценарии на этапе подтверждения

### Сценарий А: Автоматическое подтверждение
Поставщик мгновенно подтверждает бронь через API.  
→ Статус брони меняется на «Подтверждено».  
→ Система устанавливает Тайм-лимит оплаты.

### Сценарий Б: Ручное подтверждение
Поставщик отвечает с задержкой (offline-отель).  
→ Бронь находится в статусе «В ожидании».  
→ При получении ответа менеджер вручную меняет статус на «Подтверждено» или «Отклонено».

### Сценарий В: Отказ
Поставщик ответил отказом (нет мест, отель закрыт).  
→ Система автоматически переводит бронь в статус «Отклонена».  
→ Менеджеру предлагается повторить поиск аналогов.
